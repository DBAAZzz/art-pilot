# 数据存储与图片库 v1 方案

## 背景

Codex 生图完成后，图片会先落在 Codex 自己的目录中：

```txt
~/.codex/generated_images/<codexThreadId>/<callId>.png
```

图片路径由 Codex session JSONL 提供：

```txt
~/.codex/sessions/**/rollout-*<codexThreadId>.jsonl
```

如果 Art Pilot 长期依赖 Codex 的目录和 session 文件，会有几个问题：

- Codex 目录会持续增长，占用磁盘空间。
- 最近任务只存在 renderer state 中，应用重启后丢失。
- 前端展示依赖 Codex 内部路径，不利于后续清理、迁移和历史管理。
- 用户无法控制图片长期保存位置。

v1 的目标是让 Art Pilot 接管生成产物：Codex 输出目录只作为“临时收件箱”，Art Pilot 维护自己的图片库和历史数据库。

## 核心决策

Codex 负责生成，Art Pilot 负责持有。

```txt
Codex directory = 临时收件箱 / 原始输入来源
Art Pilot library = 长期图片资产目录 / 前端展示来源
SQLite database = 设置、历史任务、图片索引、清理状态
renderer state = 当前页面缓存，不作为持久数据源
```

生成图片必须先移动到 Art Pilot 图片库，再返回给前端展示。

前端不直接使用 Codex 原始图片路径。

## 图片库目录

默认图片库目录：

```txt
~/Pictures/Art Pilot/
```

用户可以在设置中改成自定义目录。

图片按月份和任务 ID 分层保存：

```txt
<libraryRoot>/
  2026-05/
    <jobId>/
      0001.png
      0002.png
```

示例：

```txt
~/Pictures/Art Pilot/
  2026-05/
    a83ca428-1f2f-44a4-a0a9-5bb8be5ad6f3/
      0001.png
```

不直接沿用 Codex 的 `ig_xxx.png` 作为图片库文件名。Codex 原始文件名可以保存在数据库中，用于排错和审计。

## 移动策略

产品语义是“移动”，不是复制保留两份。

实现上必须做安全移动，不能只依赖 `fs.rename`。

原因：

- 用户可能把图片库放在外接盘、网盘目录或不同 volume。
- 跨 volume `rename` 可能失败并返回 `EXDEV`。
- 直接删除源文件有丢图风险。

推荐 `safeMoveImage(sourcePath, targetPath)` 流程：

```txt
1. 确认 sourcePath 存在且是普通文件。
2. 创建 targetPath 所在目录。
3. 优先尝试 rename(sourcePath, targetPath)。
4. 如果 rename 失败，使用 copy + verify + unlink：
   sourcePath -> targetPath.tmp-<random>
   校验临时文件大小与源文件一致
   rename 临时文件为 targetPath
   unlink sourcePath
5. 返回 targetPath。
```

如果目标文件已存在，不覆盖。生成新的版本名：

```txt
0001.png
0001-2.png
0001-3.png
```

如果删除 Codex 源文件失败，但 Art Pilot 目标文件已经写入成功，任务不应失败。记录 cleanup error，后续由清理任务重试。

## 生成链路

当前 v2 生图链路：

```txt
Codex 退出
  -> recoverImagesFromCodexSessions
  -> 得到 Codex 图片路径
  -> generatedImageRegistry 注册 Codex 路径
  -> renderer 收到 image-found
```

数据存储与图片库 v1 应调整为：

```txt
Codex 退出
  -> recoverImagesFromCodexSessions
  -> 得到 Codex 图片路径
  -> ImageLibraryService.safeMoveImageToLibrary
  -> 得到 Art Pilot library 图片路径
  -> ImageHistoryService 写入任务和图片记录
  -> generatedImageRegistry 注册 library 路径
  -> renderer 收到 image-found
```

也就是说，`image-found.imagePath` 应该是 Art Pilot 图片库路径，而不是 Codex 原始路径。

如果保留 2 秒恢复 timer 来尽早展示图片，也应在 timer 中先移动图片，再发送 `image-found`。

## 数据库

使用 SQLite 作为本地持久化数据库。

数据库位置：

```txt
app.getPath('userData')/art-pilot.sqlite
```

macOS 示例：

```txt
~/Library/Application Support/Art Pilot/art-pilot.sqlite
```

推荐使用 `better-sqlite3`：

- 本地单文件。
- 事务可靠。
- 查询最近任务和图片列表简单。
- 适合 Electron 主进程。

注意：`better-sqlite3` 是 native dependency，Electron 打包时需要 rebuild。

## 表结构

### `settings`

保存用户配置。

```sql
CREATE TABLE settings (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL,
  updated_at INTEGER NOT NULL
);
```

建议配置项：

```txt
imageLibraryPath
codexImageCleanup
codexSessionCleanup
```

`codexImageCleanup` 可选值：

```txt
after-import
never
```

v1 默认值：

```txt
after-import
```

`codexSessionCleanup` 可选值：

```txt
after-import
after-days
never
```

如果非常关注磁盘空间，可以默认 `after-import`。但要明确它会影响 Codex 自身历史、用量统计和问题排查。

### `generation_tasks`

保存任务级数据。

```sql
CREATE TABLE generation_tasks (
  id TEXT PRIMARY KEY,
  codex_thread_id TEXT,
  prompt TEXT NOT NULL,
  count INTEGER NOT NULL,
  size TEXT,
  status TEXT NOT NULL,
  error TEXT,
  created_at INTEGER NOT NULL,
  completed_at INTEGER
);
```

`status` 可选值：

```txt
running
complete
error
cancelled
```

### `generated_images`

保存图片级数据。

```sql
CREATE TABLE generated_images (
  id TEXT PRIMARY KEY,
  task_id TEXT NOT NULL,
  image_index INTEGER NOT NULL,
  original_codex_path TEXT,
  library_path TEXT NOT NULL,
  file_size INTEGER,
  width INTEGER,
  height INTEGER,
  cleanup_status TEXT NOT NULL DEFAULT 'pending',
  cleanup_error TEXT,
  created_at INTEGER NOT NULL,
  moved_at INTEGER NOT NULL,
  FOREIGN KEY(task_id) REFERENCES generation_tasks(id)
);
```

`original_codex_path` 用于排错和清理记录。

`library_path` 是 Art Pilot 的长期图片路径，也是前端展示和历史记录的真实来源。

## 服务分层

建议新增服务：

```txt
apps/desktop/electron/services/
  databaseService.ts
  settingsService.ts
  imageLibraryService.ts
  imageHistoryService.ts
  codexCleanupService.ts
```

### `DatabaseService`

职责：

- 打开 SQLite 数据库。
- 执行 migrations。
- 提供事务能力。

### `SettingsService`

职责：

- 读取图片库路径。
- 更新图片库路径。
- 读取 Codex 清理策略。

### `ImageLibraryService`

职责：

- 根据 `jobId`、`index`、源文件扩展名生成目标路径。
- 执行安全移动。
- 返回 library 路径和文件 metadata。

接口示意：

```ts
type ImportedImage = {
  imageId: string
  index: number
  originalCodexPath: string
  libraryPath: string
  fileSize: number
  width?: number
  height?: number
}

class ImageLibraryService {
  moveImageToLibrary(input: {
    jobId: string
    index: number
    sourcePath: string
    createdAt: number
  }): Promise<ImportedImage>
}
```

### `ImageHistoryService`

职责：

- 创建任务记录。
- 更新任务状态。
- 保存图片记录。
- 查询最近任务。

最近任务不再由 React state 独占维护，而是从 SQLite 查询。

### `CodexCleanupService`

职责：

- 删除已成功移动的 Codex 原始图片。
- 按策略删除 Codex session JSONL。
- 记录清理失败原因，允许后续重试。

清理必须在图片移动和数据库写入成功后执行。

## IPC

建议新增 IPC：

```txt
settings:get
settings:update
image-history:list-recent
image-history:open-library-folder
```

已有生图 IPC 继续保留：

```txt
image:generate-start
image:generation-event
image:cancel
```

renderer 不直接访问 SQLite，也不直接读写图片库目录。

## 前端状态

ImageGeneration 页面启动时：

```txt
window.api.listRecentImageTasks()
  -> 渲染最近任务
```

生成过程中：

```txt
image-found
  -> 使用 imageUrl 展示
  -> imagePath 是 library_path
```

任务完成后：

```txt
complete
  -> 当前任务状态更新
  -> 最近任务已经在数据库中有持久记录
```

应用重启后，最近任务从 SQLite 恢复。

## 失败处理

### 移动失败

如果某张图片移动失败：

- 该图片不发送 `image-found`。
- 记录 task error 或 partial error。
- 如果至少一张图片移动成功，可以允许任务 `complete`，但 UI 应显示实际图片数量。

### 数据库写入失败

如果图片已经移动成功，但数据库写入失败：

- 不删除 Codex session。
- 不删除 Codex 源图。
- 返回错误，保留目标图片路径到日志，方便人工恢复。

### Codex 清理失败

如果图片已经移动成功、数据库写入成功，但 Codex 源图或 session 删除失败：

- 任务仍然完成。
- `cleanup_status` 标记为 `failed`。
- 后续后台清理重试。

## 清理策略

v1 支持强清理磁盘空间，但顺序必须严格：

```txt
1. 图片移动到 Art Pilot library 成功。
2. 图片 metadata 写入 SQLite 成功。
3. generatedImageRegistry 注册 library 路径成功。
4. renderer 可以通过 imageUrl 展示 library 图片。
5. 删除 Codex 原始图片。
6. 按配置删除 Codex session JSONL。
```

如果第 5 或第 6 步失败，不回滚任务。

## 迁移步骤

1. 新增 SQLite 依赖和 `DatabaseService`。
2. 新增 `settings`、`generation_tasks`、`generated_images` migrations。
3. 新增 `SettingsService`，提供默认图片库路径。
4. 新增 `ImageLibraryService.safeMoveImageToLibrary`。
5. 修改 `ImageGenerationService`：恢复 Codex 图片后先移动到 library，再注册和发送事件。
6. 新增 `ImageHistoryService`：任务开始、图片移动、任务完成都写库。
7. 修改 `generatedImageRegistry`：注册 library 路径。
8. 新增最近任务读取 IPC。
9. 修改 renderer：页面启动时读取最近任务。
10. 新增 `CodexCleanupService`，在导入成功后删除 Codex 原始图片和可选 session。

## 最终结论

数据存储与图片库 v1 的核心是把 Codex 目录从“数据源”降级为“临时收件箱”。

Art Pilot 的长期数据源应该是：

```txt
SQLite task/image metadata
Art Pilot library image files
```

前端和历史记录只依赖 Art Pilot 自己的图片库，不依赖 Codex 的内部目录结构。
